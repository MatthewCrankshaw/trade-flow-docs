# v1.8 Restructure Decisions

**Date:** 2026-04-11
**Session:** `/gsd-discuss-phase 41` → pivoted to milestone restructure
**Outcome:** Phase 41 dissolved; milestone shrinks from 8 phases to 7.

## Context

Phase 41 was originally scoped as three foundation refactors that must land before any estimate code is written:
- FND-01: Rename `quote-token` → `document-token` (unified module).
- FND-02: Rename `quote-settings` → `document-settings` (unified settings API).
- FND-03: Widen `quote_line_items` with `parentType` discriminator and optional `estimateId`.

During the discuss-phase pass for Phase 41 the user overrode two of the three research recommendations in favour of stricter separation of concerns between quote and estimate document types, which dissolved the phase.

## Decisions

### D-01: Line items — separate collections and modules

**Decision:** Do not widen `quote_line_items`. Introduce a new `estimate_line_items` collection with its own module (`EstimateLineItem*` entity, repository, creator, policy, DTOs, validators) mirroring the quote line-item stack.

**Rationale:** User explicitly prefers maintainability isolation over DRY code reuse. Even though the data shape is near-identical, keeping two collections means bug fixes in one cannot regress the other, and each document type can evolve independently without conditional branching.

**Rejected alternative:** `.planning/research/ARCHITECTURE.md §2` recommended the shared-collection-with-discriminator approach on code-reuse grounds. Overridden.

**Impact:**
- `quote_line_items` collection is **untouched forever** — no `parentType` field, no `estimateId` field, no migration, no backfill.
- Bundle, tax-rate, and pricing factories are **duplicated** for estimates. Bug fixes now touch two places.
- Phase 48 (Convert to Quote) copies line items *across* collections, not within one.
- Frontend shared components take `documentType` as a required prop and branch internally (per `.planning/research/PITFALLS.md §Pitfall 9` guidance).

### D-02: Tokens — unified module

**Decision:** Proceed with the research recommendation. `quote-token` → `document-token` module rename with `documentType: "quote" | "estimate"` discriminator and `quoteId` → `documentId` field rename. One collection (`document_tokens`), one guard (`DocumentSessionAuthGuard`), one session/rate-limit/revocation/`firstViewedAt` code path.

**Rationale:** The token entity is a single concept — an opaque customer-facing handle to a document. The guard encodes rate limiting, expiry, revocation, and view tracking that would be dangerous to duplicate (two parallel bugs, two parallel audit surfaces). Customer-facing security infrastructure is the single most-scrutinised part of the system and should have one canonical implementation.

**Impact:**
- One-shot migration: `db.quote_tokens.updateMany({}, { $set: { documentType: "quote" }, $rename: { quoteId: "documentId" } })` then `renameCollection("document_tokens")`. Reversible.
- Existing quote tokens must continue to validate after rename (verified via integration test).
- Public quote page endpoint (`/v1/public/quote/:token`) keeps its URL — internal module rename only.
- This work is **absorbed into the new Phase 41 (formerly Phase 42: Estimate Module CRUD)** and ships in the same PR as the first batch of estimate backend code.

### D-03: Settings — separate module

**Decision:** Do not rename or widen `quote-settings`. Introduce a new `estimate-settings` module with its own API surface when the estimate email template is first needed (in the new Phase 44, formerly Phase 45: Email & Send Flow). The frontend Business > Templates tab will call two APIs — one for `quoteEmailTemplate`, one for `estimateEmailTemplate`.

**Rationale:** Same separation-of-concerns preference as line items. User values isolation of per-document-type settings over single-API convenience, accepting the frontend cost of two calls on the Templates tab.

**Rejected alternative:** `.planning/research/ARCHITECTURE.md §10` recommended renaming `quote-settings` → `document-settings` with both template fields under one API. Overridden.

**Impact:**
- `quote-settings` module is **untouched forever**. No rename, no field additions.
- New `estimate-settings` module (module, controller, service, repository, policy, DTOs, requests, responses, tests) is built in the new Phase 44.
- Business > Templates tab in `trade-flow-ui` gains a second API call — fetches and saves both template types in parallel.
- SND-02 requirement is rewritten to reference `estimate-settings` as a new module, not a rename.

### D-04: Milestone restructure

**Decision:** Dissolve Phase 41 entirely. Absorb the document-token rename into the new Phase 41 (formerly Phase 42). Absorb the new `estimate-settings` module into the new Phase 44 (formerly Phase 45). Renumber Phases 42–48 down to 41–47.

**Rationale:** With FND-02 and FND-03 dissolved, Phase 41 had only the document-token rename left — too small to justify a dedicated phase when the rename naturally precedes the first estimate repository/service that needs to issue tokens (in Phase 42's work).

**Tradeoff acknowledged:** The user was warned that folding a refactor of live customer-facing token code into the same PR as new estimate module work makes regressions harder to bisect. The user chose to proceed anyway.
