## onboarding-skips-step-1 — Onboarding wizard skips step 1 and later duplicates business/trial requests under StrictMode
- **Date:** 2026-04-10
- **Error patterns:** onboarding, wizard, step 1, step 2, initialStep, useOnboardingStep, JwtAuthGuard, auth.guard, user.name, email fallback, SetupLoadingScreen, executeSetup, navigate, dashboard, OnboardingGuard, useCurrentBusiness, RTK Query, stale cache, isLoading, isFetching, invalidatesTags, refetchUser, refetchBusinesses, SubscriptionTrialCreator, startTrial, POST /v1/subscription/trial, E11000, duplicate key, subscriptions.userId, upsertByStripeCustomerId, findByUserId, find-then-insert, non-atomic, 500 Internal Server Error, Stripe customer, leaked customer, React StrictMode, double-invocation, useEffect, hasRunRef, useRef guard, POST /v1/business, duplicate business, businessRoles
- **Root cause:** Four chained defects in the registration → onboarding → dashboard flow: (1) JwtAuthGuard.canActivate defaulted a new user's `name` to `userInfo.email` when Firebase displayName was absent, so useOnboardingStep saw hasName=true and started the wizard at step 2; (2) SetupLoadingScreen called navigate("/dashboard") synchronously after createBusiness/startTrial resolved, before the RTK Query tag-invalidation refetch of /v1/user/me landed, so OnboardingGuard + useCurrentBusiness read stale cached businessRoles=[] (isLoading is false during background refetches) and redirected back to /onboarding where the wizard remounted on step 2; (3) SubscriptionTrialCreator.create used a non-atomic findByUserId-then-upsertByStripeCustomerId sequence that collided with the unique index on subscriptions.userId, throwing raw E11000 → HTTP 500 whenever two calls raced; (4) React StrictMode's intentional double-invocation of useEffect in dev mode fired executeSetup twice in parallel, producing two concurrent POST /v1/business calls (no user-scoped uniqueness on businesses, so both succeeded and two businesses were attached to the user) AND two concurrent POST /v1/subscription/trial calls (which triggered defect #3 until it was fixed).
- **Fix:** (1) Changed the newUser name fallback in trade-flow-api/src/auth/auth.guard.ts from `userInfo.name || userInfo.displayName || userInfo.email` to `userInfo.name || userInfo.displayName || null`. (2) In trade-flow-ui/src/features/onboarding/components/SetupLoadingScreen.tsx, executeSetup now awaits `Promise.all([refetchUser(), refetchBusinesses()])` after startTrial resolves and only calls navigate("/dashboard") once the refetched user has at least one businessRole; otherwise it surfaces the error state. (3) In trade-flow-api/src/subscription/services/subscription-trial-creator.service.ts, made SubscriptionTrialCreator.create idempotent (returns existing sub if findByUserId finds one) and added belt-and-braces E11000 recovery: on duplicate-key error during upsert, re-fetch by userId and return that row instead of propagating the Mongo error. (4) In trade-flow-ui/src/features/onboarding/components/SetupLoadingScreen.tsx, added a `hasRunRef = useRef(false)` guard to the mount useEffect so executeSetup runs exactly once per component mount regardless of StrictMode's synchronous unmount/remount cycle (refs persist across it; state does not). Added a regression test that wraps SetupLoadingScreen in <StrictMode> and asserts createBusiness and startTrial are each called exactly once. Known follow-up: the defect #3 E11000 recovery path still leaks one Stripe customer when a genuine network-retry race occurs in production.
- **Files changed:** trade-flow-api/src/auth/auth.guard.ts, trade-flow-ui/src/features/onboarding/components/SetupLoadingScreen.tsx, trade-flow-ui/src/features/onboarding/components/__tests__/SetupLoadingScreen.test.tsx, trade-flow-api/src/subscription/services/subscription-trial-creator.service.ts, trade-flow-api/src/subscription/test/services/subscription-trial-creator.service.spec.ts
