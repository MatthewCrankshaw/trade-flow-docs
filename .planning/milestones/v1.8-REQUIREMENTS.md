# Requirements Archive: v1.8 Estimates

**Archived:** 2026-04-18
**Milestone:** v1.8 Estimates
**Status:** ✅ SHIPPED
**Original defined:** 2026-04-11

## v1.8 Requirements (All Complete)

### Estimate Core

- [x] **EST-01**: User can create an estimate via a document-type toggle (Quote / Estimate) on the shared create dialog
- [x] **EST-02**: Estimates are numbered E-YYYY-NNN via a new atomic `estimate_counters` collection
- [x] **EST-03**: Estimate creation uses its own dedicated `estimate_line_items` collection and module
- [x] **EST-04**: User can edit an estimate in Draft status (scope, line items, contingency, notes, customer, job)
- [x] **EST-05**: User can soft-delete an estimate from Draft status (DELETED enum; preserves line item history)
- [x] **EST-06**: User can view a list of all estimates with tab filtering by status
- [x] **EST-07**: User can view estimate detail with line items, contingency range, totals, status, customer info, and response summary
- [x] **EST-08**: Estimate has status lifecycle: Draft -> Sent -> Viewed -> Responded -> (Converted / Declined / Expired / Lost) with valid transitions enforced by API
- [x] **EST-09**: `quote-token` module is renamed to `document-token` with `documentType` discriminator

### Contingency & Display

- [x] **CONT-01**: User can set contingency percentage via a slider in the estimate form (0-30% in 5% increments, default 10%)
- [x] **CONT-02**: Estimate displays price as a range (£X-£Y) by default
- [x] **CONT-03**: User can toggle estimate price display to "From £X" mode
- [x] **CONT-04**: User can add optional uncertainty reasons via five quick-tap preset chips plus freeform notes
- [x] **CONT-05**: Contingency range math is computed API-side only

### Revisions

- [x] **REV-01**: Estimate entity stores `parentEstimateId`, `revisionNumber`, and `isCurrent`
- [x] **REV-02**: User can revise a Sent estimate via an "Edit and resend" action (Phase 49 gap closure)
- [x] **REV-03**: Only one revision per estimate chain can have `isCurrent: true`, enforced by partial unique index
- [x] **REV-04**: Estimate detail shows a collapsed History section listing previous revisions (Phase 49 gap closure)
- [x] **REV-05**: Revising an estimate cancels pending follow-ups and schedules a fresh sequence

### Send & Email

- [x] **SND-01**: User can send an estimate to a customer via email with a secure public link
- [x] **SND-02**: A new `estimate-settings` module exposes `estimateEmailTemplate` via its own API
- [x] **SND-03**: Send dialog allows the user to review and edit the pre-filled subject and rich-text body
- [x] **SND-04**: Rendered email HTML is persisted on the estimate at send time for audit trail
- [x] **SND-05**: Default estimate email template includes mandatory non-binding legal language
- [x] **SND-06**: User can re-send an estimate already in Sent or revised status without creating a new revision
- [x] **SND-07**: Estimate email subject line includes "Estimate" (not "Quote")

### Customer-Facing Public Page

- [x] **CUST-01**: Customer can view the full estimate via a secure token link without logging in
- [x] **CUST-02**: Customer page shows estimate details: scope, price range, contingency explanation, validity, uncertainty notes
- [x] **CUST-03**: Customer page displays non-binding legal language prominently
- [x] **CUST-04**: Customer page has NO "Accept" button, NO digital signature, NO single fixed total
- [x] **CUST-05**: Customer page records `firstViewedAt` and triggers status transition to Viewed
- [x] **CUST-06**: Customer page always resolves to the latest revision in the estimate chain
- [x] **CUST-07**: Customer page sets zero non-essential cookies (PECR compliance)

### Customer Response Handling

- [x] **RESP-01**: Customer can tap "Happy to Proceed" to signal intent; status transitions to RESPONDED
- [x] **RESP-02**: Customer can tap "Message [Tradesperson First Name]" to send a freeform message
- [x] **RESP-03**: Customer can tap "Not right for me" to decline with structured reason
- [x] **RESP-04**: After response, action area replaced with inline success confirmation
- [x] **RESP-05**: Trader receives email notification on any customer response
- [x] **RESP-06**: Estimate status automatically transitions on customer response
- [x] **RESP-07**: Terminal-state estimates show friendly read-only message with no active buttons
- [x] **RESP-08**: Customer response data persisted for trader's detail view (Phase 50 gap closure)

### Follow-up Automation

- [x] **FUP-01**: Sending an estimate schedules 3-day, 10-day, and 21-day follow-up emails via BullMQ
- [x] **FUP-02**: Follow-up delays are relative in UTC for DST safety
- [x] **FUP-03**: Follow-up jobs use deterministic `jobId` pattern for idempotent add and O(1) cancellation
- [x] **FUP-04**: Each follow-up email includes the estimate summary and three response actions
- [x] **FUP-05**: Any exit transition cancels all pending follow-ups for that estimate chain
- [x] **FUP-06**: Worker processor performs defence-in-depth state check before sending
- [x] **FUP-07**: Estimates auto-transition to Expired 30 days after Sent timestamp
- [x] **FUP-08**: Production Redis has AOF persistence enabled for scheduled follow-up durability

### Convert to Quote

- [x] **CONV-01**: User can convert a Sent or Responded estimate to a new quote
- [x] **CONV-02**: Convert copies line items with literal tax rate snapshots, drops contingency
- [x] **CONV-03**: Convert opens new quote in edit mode for mandatory review (Phase 50 gap closure)
- [x] **CONV-04**: Convert endpoint accepts Idempotency-Key header for double-submit protection
- [x] **CONV-05**: Converted estimate transitions to Converted status with `convertedToQuoteId` back-link
- [x] **CONV-06**: Converted quote displays "Converted from E-YYYY-NNN" back-link

### Mark as Lost

- [x] **LOST-01**: User can manually mark an estimate as Lost with a structured reason or freeform text
- [x] **LOST-02**: Marking as Lost cancels all pending follow-ups
