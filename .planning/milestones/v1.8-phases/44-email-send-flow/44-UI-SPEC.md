## SendEstimateDialog Layout Contract

### Dialog sizing

`sm:max-w-lg` (medium dialog -- standard form with 3 fields including a rich text area). Identical to SendQuoteDialog.

### Structure

```
+------------------------------------------+
|  Send Estimate                      [x]  |
|  Review and send this estimate to your   |
|  customer.                               |
|                                          |
|  To                                      |
|  [john@example.com                    ]  |
|  ! No email on file for this customer    |
|  [ ] Save email to customer record       |
|                                          |
|  Subject                                 |
|  [Estimate E-2026-001 for Boiler Inst ]  |
|  ! Subject must contain the word         |
|    "Estimate"                            |
|                                          |
|  Message                                 |
|  +--------------------------------------+|
|  | B  I  Link  |                        ||
|  |--------------------------------------|
|  | Hi John,                             ||
|  |                                      ||
|  | Please find your estimate E-2026-001 ||
|  | for Boiler Installation. This is a   ||
|  | guide price -- a firm quote will     ||
|  | follow after a site visit.           ||
|  |                                      ||
|  | Kind regards,                        ||
|  | Mike                                 ||
|  +--------------------------------------+|
|                                          |
|         [Keep Editing]  [Send Estimate]  |
+------------------------------------------+
```

### Field specifications

| Field | Type | Pre-fill source | Validation |
|-------|------|----------------|------------|
| To | Email input (`Input`) | `customer.email` or empty if null | Required, valid email format |
| Subject | Text input (`Input`) | Default template subject with variables resolved; must contain "Estimate" (case-insensitive) | Required, non-empty, must contain "Estimate" |
| Message | Rich text (`RichTextEditor`) | Default template body with variables resolved | Required, non-empty |

### Template variables resolved client-side (before display)

`{{estimateNumber}}`, `{{jobTitle}}`, `{{customerName}}`, `{{businessName}}`, `{{userName}}`

Source: CONTEXT.md D-LGL-06.

### Auto-focus

Focus lands on the message body (Tiptap editor) when the dialog opens. Use `autofocus: "end"` on the Tiptap editor instance.

### Warning state (no customer email)

When `customer.email` is null:
- To field renders empty
- Below To field: `Alert` component with `warning` variant -- icon `AlertTriangle` (`h-4 w-4`) + text "No email on file for this customer"
- Below alert: Checkbox row -- `[ ] Save email to customer record` (`text-sm font-semibold`)
- Checkbox always visible when To field has a value (regardless of whether customer had an email), unchecked by default, checked by default when customer had no email

### Subject validation inline error

When the user edits the subject to remove "Estimate":
- Below Subject field: inline error text in `text-sm text-destructive` -- "Subject must contain the word 'Estimate'"
- Send button remains enabled (server-side is authoritative per D-DOC-05); the inline hint is UX courtesy

Source: CONTEXT.md D-LGL-05, D-UI-01.

### Send button states

| State | Button |
|-------|--------|
| Ready | `<Button>Send Estimate</Button>` (default variant) with `Send` icon (lucide) `h-4 w-4` |
| Submitting | `<Button disabled>Sending...</Button>` with loading spinner |
| To field empty | `<Button disabled>Send Estimate</Button>` |

### Post-send behavior

1. `toast.success("Estimate sent to john@example.com")` -- includes the actual recipient email
2. Dialog closes via `onOpenChange(false)`
3. RTK Query cache invalidation refreshes estimate detail -- status badge updates to "Sent", `lastSentAt` timestamp appears

Source: CONTEXT.md D-UI-04 (tag invalidation pattern).

## State Contracts

### Loading State (SendEstimateDialog)

Dialog opens immediately with fields pre-filled from local data (customer email, resolved template). No loading skeleton needed -- all data is available from the already-loaded estimate detail page context.

If business settings (template) have not been fetched yet, use the system default template as fallback.

### Loading State (EstimateEmailSettings)

Standard skeleton/loading state while `useGetEstimateSettingsQuery` is in flight. Fields render empty with placeholder text showing system defaults.

### Error States

| Condition | Display | Copy |
|-----------|---------|------|
| Resend delivery failure | Toast error | `toast.error("Failed to send estimate. Please try again.")` |
| Network error on send | Toast error | `toast.error("Could not connect to the server. Check your connection and try again.")` |
| Invalid email (server-side) | Toast error | `toast.error("The email address is not valid. Please check and try again.")` |
| Subject missing "Estimate" (server-side) | Toast error + inline | `toast.error("Estimate email subject must contain the word 'Estimate'.")` + inline error below Subject field |
| Estimate not sendable (terminal status) | Toast error | `toast.error("This estimate cannot be sent in its current status.")` |
| Business settings save failure | Toast error | `toast.error("Failed to save template. Please try again.")` |

All errors are toast-based. The dialog remains open on send failure so the user can retry without re-entering data.

### Empty States

| Location | Condition | Display |
|----------|-----------|---------|
| Estimate settings template fields | Fields are empty (no template configured) | Placeholder text in input/textarea showing the system default values in `text-muted-foreground` |

No full empty state illustration needed -- the settings form is always visible with placeholder defaults.
