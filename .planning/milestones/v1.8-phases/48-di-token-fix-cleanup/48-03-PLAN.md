```ts

After:

const CONVERTIBLE_STATUSES: readonly EstimateStatus[] = ["sent", "responded"];

```

Note three things:
1. `"site_visit_requested"` is REMOVED from `REVISABLE_STATUSES`.
2. The `as const` suffix is REMOVED — no longer needed because each tuple is explicitly typed as `readonly EstimateStatus[]`, which gives the same readonly guarantee while enforcing that every literal is a member of the `EstimateStatus` union.
3. If any future edit adds a literal outside the union (e.g. a typo or a removed status), `tsc` will reject it at compile time — restoring the compiler guard that was missing during Plan 02.

**Change 3 — remove widening casts at call sites (lines 50, 51, 53):**

Before:
```ts
const canConvert = (CONVERTIBLE_STATUSES as readonly string[]).includes(estimate.status);
const canMarkLost = (LOSABLE_STATUSES as readonly string[]).includes(estimate.status);
const canResend = ["sent", "viewed", "responded"].includes(estimate.status);
const canRevise = (REVISABLE_STATUSES as readonly string[]).includes(estimate.status);
```

After:
```ts
const canConvert = CONVERTIBLE_STATUSES.includes(estimate.status);
const canMarkLost = LOSABLE_STATUSES.includes(estimate.status);

const canRevise = REVISABLE_STATUSES.includes(estimate.status);
```

The casts are no longer needed because `estimate.status` is `EstimateStatus` and the tuples are now `readonly EstimateStatus[]` — the element types match, so `.includes()` type-checks natively.

Note: Line 52 (`canResend`) is an inline array literal, not one of the three named constants. Leave it alone — it is not within the scope of this gap (VERIFICATION.md flagged only the three named constants at lines 50, 51, 53). Widening is fine there because all three literals are valid `EstimateStatus` members and `.includes(string)` narrows correctly from an inline literal.
