## Overview

Prepared trade-flow-api as a monorepo with two independently deployable NestJS services -- the existing API and a new background worker -- connected via BullMQ/Redis. The dual entry-point pattern (preserving all path aliases) was chosen over NestJS CLI monorepo mode to avoid webpack and a massive migration.

## Milestone Summary

**Key Decisions:**

- Dual entry-point pattern over NestJS CLI monorepo mode -- preserves 20+ path aliases, avoids webpack
- Redis `maxmemory-policy noeviction` required from day one -- BullMQ silently loses jobs otherwise
- Worker uses `createApplicationContext()` -- no HTTP server needed for background processing
- No Redis volume -- ephemeral storage; transient BullMQ jobs don't need persistence in dev
- IORedis instance cast via `as unknown as` to resolve type mismatch between top-level ioredis and BullMQ bundled ioredis
- CoreModule included in WorkerModule -- MongoConnectionService requires MONGO_URL even in worker context
- `deleteOutDir:false` in worker-cli.json -- prevents clobbering dist/main.js during worker build
- Debug port 9230 for worker to avoid collision with API on 9229
- Worker reuses development Dockerfile stage -- no separate worker-specific dev stage needed

**Issues Resolved:**

- Path aliases resolved for all three compile targets (tsc, Jest, NestJS build) with a single tsconfig.json entry
- IORedis type mismatch between top-level package and BullMQ bundled version resolved via type cast

**Issues Deferred:**

- Real job processors (email, PDF) -- v1.4 is infrastructure only; echo processor proves the pipeline
- Bull Board admin UI -- defer until real jobs exist
- Monorepo directory restructure (API + worker as sibling services with shared `libs/`) -- dual entry-point pattern ships instead; restructure deferred to future milestone
- Shared module pattern for worker importing API services -- worker imports `@core/*` directly; formal shared modules deferred

**Technical Debt Incurred:**

- Echo processor is test-only -- real processors will be added in future milestones (email, PDF, billing)
- POST /v1/queue/test-echo is a diagnostic endpoint only -- should be removed or auth-guarded before production
