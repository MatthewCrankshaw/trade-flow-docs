Guard canActivate logic (subscription.guard.ts):
1. Check isPublic metadata -- if true, return true
2. Check skipSubscriptionCheck metadata -- if true, return true
3. If HTTP method is GET or HEAD -- return true
4. If user has supportRoles -- return true
5. DB lookup: subscriptionRepository.findByUserId(user.externalAuthUserId)
6. If no subscription OR status not TRIALING/ACTIVE -- throw ForbiddenError
7. Return true

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> API | HTTP requests cross from untrusted browser to API; JwtAuthGuard validates identity |
| subscription decorator -> guard | Metadata on controller methods signals guard to skip subscription check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-01 | Elevation of Privilege | UserController.patch() with @SkipSubscriptionCheck | accept | JwtAuthGuard still enforces authentication. PATCH /v1/user/me only updates the authenticated user's own record (self-only by design). An expired-subscription user updating their name is low-risk and functionally identical to onboarding. |
| T-40-02 | Elevation of Privilege | BusinessController.create() with @SkipSubscriptionCheck | accept | JwtAuthGuard enforces authentication. Existing BusinessPolicy enforces one-business-per-user constraint. A user cannot create multiple businesses to abuse this bypass. |
| T-40-03 | Tampering | Decorator removal regression | mitigate | Unit tests in subscription.guard.spec.ts verify bypass behavior. Removing decorator causes test failure. |
</threat_model>
