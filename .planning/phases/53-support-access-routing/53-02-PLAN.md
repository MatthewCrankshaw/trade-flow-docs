   - First check `location.state?.from` -- if present, navigate there (per D-03, honours ProtectedRoute's saved location)
   - Then check `user.supportRoles.length > 0` -- if true, navigate to `/support` (per D-01, D-02)

   The implementation approach depends on how LoginPage currently works. Two patterns:

   **Pattern A (if handleSignInSuccess is sync callback):**
   The role-aware redirect cannot run inside handleSignInSuccess because user data from RTK Query may not be available yet. Instead:
   - In `handleSignInSuccess`, set a local state flag (e.g., `setLoginSucceeded(true)`)
   - Use `useGetCurrentUserQuery(undefined, { skip: !loginSucceeded })` to fetch user data only after login succeeds
   - Add a `useEffect` that watches for `isSuccess && user`:
     ```typescript
     useEffect(() => {

       const savedLocation = (location.state as { from?: string })?.from;
       if (savedLocation) {
         navigate(savedLocation, { replace: true });
         return;
       }
       if (user.supportRoles.length > 0) {
         navigate("/support", { replace: true });
       } else {
         navigate("/dashboard", { replace: true });
       }
     }, [isSuccess, user, navigate, location.state]);
     ```

   **Pattern B (if auth state is already reactive):**
   If the login page already has access to user data via the auth context or RTK Query (check existing code), integrate the role check directly.


<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client route guard (SupportGuard) | UX convenience only; backend API (Phase 52 PermissionGuard) is authoritative |
| Login redirect logic | Client-side routing decision based on server-provided role data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-53-01 | Tampering | SupportGuard client-side bypass via URL manipulation | accept | Frontend guards are UX convenience only; backend API enforces permissions via Phase 52 PermissionGuard. Direct navigation to /support without support role results in redirect to /dashboard, not data exposure. |
| T-53-02 | Elevation of Privilege | Support role spoofing in client state | accept | Role data from useGetCurrentUserQuery is server-authoritative; RTK Query cache reflects server state. API endpoints enforce permissions independently of client routing. |
| T-53-03 | Spoofing | Manipulated location.state.from to redirect to arbitrary URL | mitigate | React Router Navigate with replace:true uses client-side routing only (same-origin SPA navigation). location.state.from cannot redirect to external URLs -- it is a path string consumed by useNavigate, which only handles internal routes. |
| T-53-04 | Information Disclosure | Stale role data after role revocation shows support nav | accept | RTK Query refetches user on page load/token refresh. Phase 55 (RADM-04) addresses immediate effect of role changes. Acceptable risk for this phase -- no sensitive data in support nav items themselves. |
</threat_model>
