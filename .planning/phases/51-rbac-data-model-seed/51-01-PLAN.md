Create `permission.repository.ts` following the SupportRoleRepository pattern exactly, with these additions for in-memory caching (per RESEARCH.md pitfall 3 -- avoid N+1 query on every request):

- `private static readonly COLLECTION = "permissions";`
- `private readonly logger = new AppLogger(PermissionRepository.name);`
- `private permissionCache: Map<string, IPermissionDto> = new Map();`
- Constructor takes `MongoDbFetcher` dependency (same as SupportRoleRepository)
- `findByIds(ids: string[]): Promise<DtoCollection<IPermissionDto>>` -- check cache first. If all IDs are in cache, return from cache without DB call. If any are missing, fetch from DB and update cache. Return DtoCollection using same `toDtoCollection` pattern as SupportRoleRepository.
- `findAll(): Promise<DtoCollection<IPermissionDto>>` -- fetch all permissions, populate cache. Used by seeder on boot.
- `private mapToDto(entity: PermissionEntity): IPermissionDto` -- maps `_id.toString()` to `id`, copies `name`, `description`, `category`.
- `private toDtoCollection(entityCollection: EntityCollection<PermissionEntity>): DtoCollection<IPermissionDto>` -- same pattern as SupportRoleRepository.
- `clearCache(): void` -- clears the in-memory Map (useful for testing).

The cache strategy: permissions are seeded on boot and never change at runtime. The Map is populated lazily on first `findByIds` or eagerly via `findAll`. Since there are only ~19 permissions, the entire set fits trivially in memory.
