## Interaction Contracts

### Roles Card Enhancement

1. The Roles card from Phase 54 is extended with an action area below the role badges list.
2. The action area is only visible when the viewer has the `super_user` role. Admin support users see the Roles card as read-only (no buttons).
3. Button placement: below the role badges list, left-aligned, separated by `mt-4` (16px) from the badges.

### Grant Admin Role Flow

1. **Visibility**: Button appears when the target user has NO support role (customer user only). Not shown for users who already have any support role.
2. **Button**: `Button variant="default"` with `ShieldPlus` icon (16x16, `mr-2`) and label "Grant Admin Role".
3. **Click**: Opens `GrantRoleDialog` (AlertDialog).
4. **Dialog content**:
   - Title: "Grant Admin Role"
   - Description: "Grant admin access to {name}? They will be able to access the support dashboard and manage users."
   - Cancel button: "Cancel" (`AlertDialogCancel`)
   - Action button: "Grant Role" (`AlertDialogAction`, default variant)
5. **On confirm**: Calls `POST /v1/users/:id/support-role` via RTK Query mutation.
6. **On success**: Dialog closes, toast: "Admin role granted to {name}". RTK Query cache tags `['User', 'UserList']` invalidated.
7. **On error**: Dialog closes, error toast: "Unable to update role. Please try again."
8. **Loading state**: Action button shows spinner and is disabled during mutation. Cancel button is also disabled during mutation.

### Revoke Admin Role Flow

1. **Visibility**: Button appears when the target user has the `admin` support role AND the viewer is a super user AND the target is NOT the viewer themselves (D-10).
2. **Hidden for super users**: When viewing a super user's profile, no Revoke button appears at all. The super user role is system-managed (D-12).
3. **Self-protection**: When the viewer is viewing their own profile, no Revoke button appears (D-10, defense in depth).
4. **Button**: `Button variant="destructive" size="default"` with `ShieldMinus` icon (16x16, `mr-2`) and label "Revoke Admin Role".
5. **Click**: Opens `RevokeRoleDialog` (AlertDialog).
6. **Dialog content**:
   - Title: "Revoke Admin Role"
   - Description: "Remove admin access from {name}? They will lose access to support tools immediately."
   - Cancel button: "Cancel" (`AlertDialogCancel`)
   - Action button: "Revoke Role" (`AlertDialogAction`, `variant="destructive"`)
7. **On confirm**: Calls `DELETE /v1/users/:id/support-role` via RTK Query mutation.
8. **On success**: Dialog closes, toast: "Admin role revoked from {name}". RTK Query cache tags `['User', 'UserList']` invalidated.
9. **On error (generic)**: Dialog closes, error toast: "Unable to update role. Please try again."
10. **On error (self-revocation 403)**: Error toast: "You cannot revoke your own role." (backend defense in depth, should not occur due to frontend hiding).
11. **On error (last-admin 403)**: Error toast: "Cannot revoke the last super user." (backend defense in depth for super user count check).
12. **Loading state**: Action button shows spinner and is disabled during mutation. Cancel button is also disabled during mutation.

### RTK Query Integration

1. Two new mutation endpoints added to the support API slice:
   - `grantSupportRole`: `POST /v1/users/:id/support-role`
   - `revokeSupportRole`: `DELETE /v1/users/:id/support-role`
2. Both mutations invalidate cache tags: `['User']` (user detail) and `['UserList']` (user list page, so role badges update if the user navigates back).
3. No request body required for grant (the backend uses the seeded Admin role). No request body for revoke.

### Accessibility

1. All confirmation dialogs use Radix AlertDialog which provides modal focus trap and Escape key to cancel.
2. Grant button: `aria-label="Grant admin role to {name}"`.
3. Revoke button: `aria-label="Revoke admin role from {name}"`.
4. Loading spinner in buttons uses `aria-hidden="true"` with button text providing the accessible label.
