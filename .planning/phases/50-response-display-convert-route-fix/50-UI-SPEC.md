# Phase 50 -- UI Design Contract

> Visual and interaction contract for the Response Display card on the trader's EstimateDetailPage and the convert-to-quote route fix. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Applicability

Phase 50 has one new visual element and one routing fix:

1. **Response Display Card** -- The placeholder "Response handling ships in Phase 45" card on EstimateDetailPage is replaced with a card that renders actual `responseSummary` data from the API (RESP-08).
2. **Convert Route** -- A `/quotes/:quoteId/edit` route is added to App.tsx pointing to the existing `QuoteDetailPage` component (CONV-03).

The remaining two changes (backend `toDto()` derivation fix and frontend type field name alignment) are invisible to the user -- they fix data flow, not visual presentation.

## Response Card Layout Contract

### Card Structure

The response card replaces the existing placeholder at the same position in the EstimateDetailPage layout.

**Empty state (responseSummary is null):**

```
+---------------------------------------+
| Customer Response          (card title)|
|                                        |
| No customer response yet.              |
| (text-sm text-muted-foreground)        |
+---------------------------------------+
```

**Populated state (responseSummary is not null):**

```
+---------------------------------------+
| Customer Response          (card title)|
|                                        |
| [icon] Happy to Proceed    (type+label)|
| 12 Apr 2026 at 14:32       (timestamp) |
|                                        |
| "I'd like to go ahead with the work." |
| (message, if present)                  |
+---------------------------------------+
```

**Declined state (responseSummary.lastResponseType === "decline"):**

```
+---------------------------------------+
| Customer Response          (card title)|
|                                        |
| [icon] Declined            (type+label)|
| 12 Apr 2026 at 14:32       (timestamp) |
|                                        |
| Reason: Too expensive       (if reason)|
|                                        |
| "The price is higher than expected."   |
| (message, if present)                  |
+---------------------------------------+
```

### Layout Details

| Element | Classes | Notes |
|---------|---------|-------|
| Card title | `text-base` | Matches existing card titles on EstimateDetailPage |
| Response type row | `flex items-center gap-2` | Icon + label inline |
| Response type icon | `h-4 w-4` | Sized to match label text |
| Response type label | `text-sm font-semibold` | Uses declared label role |
| Timestamp | `text-xs text-muted-foreground` | Below type label, no gap needed (normal block flow with `mt-1`) |
| Decline reason label | `text-sm text-muted-foreground` | Prefixed with "Reason: " |
| Message text | `text-sm` | Rendered in `text-foreground`, block-level below timestamp/reason |
| Vertical rhythm | `flex flex-col gap-3` inside CardContent | 12px between type row, reason, and message sections |

### Response Type Label Map

| API `lastResponseType` | Display Label | Icon | Icon Color |
|------------------------|---------------|------|------------|
| `proceed` | Happy to Proceed | `ThumbsUp` | `text-green-600 dark:text-green-400` |
| `message` | Customer Message | `MessageSquare` | `text-amber-600 dark:text-amber-400` |
| `decline` | Declined | `ThumbsDown` | `text-destructive` |

### Decline Reason Label Map

| API `declineReason` | Display Label |
|---------------------|---------------|
| `too_expensive` | Too expensive |
| `going_with_someone_else` | Going with someone else |
| `decided_not_to_do_work` | Decided not to do the work |
| `just_getting_idea` | Just getting an idea of costs |
| `timing_not_right` | Timing isn't right |

Source: Phase 45 UI-SPEC decline reason chips. Same taxonomy used in MarkAsLostDialog.

## Copywriting Contract

| Element | Copy |
|---------|------|

| Proceed label | "Happy to Proceed" |

| Decline reason prefix | "Reason: {reason label}" |

No primary CTA in this phase (the response card is read-only display for the trader).
No destructive actions in this phase.
No error states specific to this card (data comes from the same estimate fetch -- if the estimate loads, the response data is included or null).
