  <action>
    Clone the JobType data model files to create the Visit Type equivalents. Key differences from JobType:

    **Entity (`IVisitTypeEntity extends IBaseEntity`):**
    - Fields: `businessId` (ObjectId), `name` (string), `description` (string | null), `color` (string), `icon` (string), `status` (VisitTypeStatus)
    - Mirror `IJobTypeEntity` structure exactly, replacing `jobType` references with `visitType`
    - Add `color` (string) and `icon` (string) fields that JobType does NOT have

    **DTO (`IVisitTypeDto extends IBaseResourceDto`):**
    - Same fields as entity but with `id: string` and `businessId: string` (string not ObjectId)
    - Mirror `IJobTypeDto` structure

    **Enum (`VisitTypeStatus`):**
    - Values: `ACTIVE = "active"`, `INACTIVE = "inactive"`
    - Clone directly from `JobTypeStatus`

    **CreateVisitTypeRequest:**
    - Fields: `name` (@IsString, @IsNotEmpty, @Length(1, 100)), `description` (@IsString, @IsOptional, nullable), `color` (@IsString, @IsNotEmpty, @Matches(/^#[0-9A-Fa-f]{6}$/)), `icon` (@IsString, @IsNotEmpty)
    - NO `status` field on create -- defaults to ACTIVE

    **UpdateVisitTypeRequest (CRITICAL: NO `name` field -- name is locked after creation):**
    - Fields: `description` (@IsString, @IsOptional, nullable), `color` (@IsString, @IsOptional, @IsNotEmpty, @Matches(/^#[0-9A-Fa-f]{6}$/)), `icon` (@IsString, @IsOptional, @IsNotEmpty), `status` (@IsEnum(VisitTypeStatus), @IsOptional)
    - This deliberately differs from UpdateJobTypeRequest which DOES allow name changes

    **Response (`IVisitTypeResponse`):**
    - Fields: `id`, `name`, `description`, `color`, `icon`, `status`
    - Independent of DTO (do not extend DTO -- duplicate fields per codebase convention)

    **Mappers (clone from job-type mappers with adaptations):**
    - `map-create-visit-type-request-to-dto.utility.ts`: Maps CreateVisitTypeRequest + businessId + generated ObjectId → IVisitTypeDto. Sets `status: VisitTypeStatus.ACTIVE` by default.
    - `map-visit-type-to-response.utility.ts`: Maps IVisitTypeDto → IVisitTypeResponse (pick relevant fields).
    - `merge-existing-visit-type-with-changes.utility.ts`: Merges UpdateVisitTypeRequest into existing IVisitTypeDto. MUST NOT overwrite `name` -- only merge `description`, `color`, `icon`, `status` when provided.

    Use `@visit-type/*` path aliases for all internal imports. Follow exact import order conventions from CONVENTIONS.md.

    Create the business logic layer, controller, module, and wire everything up.

    **Repository (`VisitTypeRepository`):**
    - Clone from `JobTypeRepository`
    - Collection name: `"visitTypes"`
    - Methods:
      - `create(dto: IVisitTypeDto): Promise<IVisitTypeDto>` -- standard create
      - `findByIdOrFail(id: string): Promise<IVisitTypeDto>` -- standard find-by-id
      - `findVisitTypesByBusinessId(businessId: string, queryOptions: IBaseQueryOptionsDto): Promise<DtoCollection<IVisitTypeDto>>` -- **MUST filter by `status: VisitTypeStatus.ACTIVE`** (this is a NEW behavior beyond JobType which does NOT filter by status)
      - `update(dto: IVisitTypeDto): Promise<IVisitTypeDto>` -- standard update
      - `countActiveByBusinessId(businessId: string): Promise<number>` -- NEW method not in JobType. Count documents where `businessId` matches AND `status: VisitTypeStatus.ACTIVE`. Use `MongoConnectionService.getDb().collection("visitTypes").countDocuments(filter)` or extend MongoDbFetcher if it has a count method. Check `MongoDbFetcher` first for existing count capability.
      - `findByBusinessIdAndName(businessId: string, name: string): Promise<IVisitTypeDto | null>` -- For checking unique name constraint at service level (complement to DB index). Query by `{ businessId: new ObjectId(businessId), name }`. Return null if not found.
    - Entity-to-DTO and DTO-to-entity conversion methods (mirror JobType pattern)

    **VisitTypeCreatorService:**
    - Clone from `JobTypeCreatorService`
    - Inject `VisitTypeRepository` and `AuthorizedCreatorFactory`
    - Method: `create(authUser: IUserDto, dto: IVisitTypeDto): Promise<IVisitTypeDto>`
    - Before creation, check for existing visit type with same name in same business via `repository.findByBusinessIdAndName()`. If found, throw `InvalidRequestError` with code `ErrorCodes.VISIT_TYPE_NAME_ALREADY_EXISTS`
    - The DB unique index is the ultimate guard, but the service-level check provides better error messages

    **VisitTypeRetrieverService:**
    - Clone from `JobTypeRetrieverService`
    - Methods: `findByIdOrFail(id)`, `findByBusinessId(businessId, queryOptions)`
    - Uses `AccessControllerFactory` with `VisitTypePolicy` for authorization

    **VisitTypeUpdaterService:**
    - Clone from `JobTypeUpdaterService`
    - Method: `update(authUser: IUserDto, existing: IVisitTypeDto, updates: IVisitTypeDto): Promise<IVisitTypeDto>`
    - **CRITICAL NEW BEHAVIOR: Last-active guard.** When `existing.status === ACTIVE` and `updates.status === INACTIVE`, call `repository.countActiveByBusinessId(existing.businessId)`. If count <= 1, throw `InvalidRequestError` with code `ErrorCodes.VISIT_TYPE_LAST_ACTIVE_CANNOT_ARCHIVE` and message "Cannot archive the last active visit type"
    - Uses `AccessControllerFactory` with `VisitTypePolicy`

    **VisitTypePolicy:**
    - Clone from `JobTypePolicy` -- extends `BasePolicy<IVisitTypeDto>`
    - Authorization checks based on user-business membership (same as JobType)

    **VisitTypeController:**
    - Clone from `JobTypeController`
    - 4 endpoints:
      - `GET /v1/business/:businessId/visit-types` -- List active visit types. Calls retriever.findByBusinessId(). Maps to response array.
      - `POST /v1/business/:businessId/visit-type` -- Create visit type. Maps request to DTO via mapper, calls creator.create(). Returns single response.
      - `GET /v1/visit-type/:visitTypeId` -- Get by ID. Calls retriever.findByIdOrFail(). Returns single response.
      - `PATCH /v1/business/:businessId/visit-type/:visitTypeId` -- Update. Gets existing via retriever, merges changes via mapper, calls updater.update(). Returns single response.
    - All endpoints use `@UseGuards(JwtAuthGuard)`
    - Follow existing try-catch + createHttpError pattern

    **VisitTypeModule:**
    - Clone from `JobModule` structure
    - Import CoreModule
    - Declare: VisitTypeController
    - Provide: VisitTypeRepository, VisitTypeCreatorService, VisitTypeRetrieverService, VisitTypeUpdaterService, VisitTypePolicy
    - Export: VisitTypeCreatorService (needed by Plan 02 for DefaultVisitTypesCreatorService)

    **Error codes (`errors-map.constant.ts`):**
    - Add two new entries to the ErrorCodes enum and errors map:
      - `VISIT_TYPE_NAME_ALREADY_EXISTS = "VISIT_TYPE_0"` → message: "A visit type with this name already exists"
      - `VISIT_TYPE_LAST_ACTIVE_CANNOT_ARCHIVE = "VISIT_TYPE_1"` → message: "Cannot archive the last active visit type"

    **Configuration:**
    - `tsconfig.json`: Add `"@visit-type/*": ["src/visit-type/*"]` to `paths` (follow pattern of `@job/*`)
    - `package.json`: Add `"^@visit-type/(.*)$": "<rootDir>/src/visit-type/$1"` to jest `moduleNameMapper` (follow pattern of `@job/*`)
    - `app.module.ts`: Add `VisitTypeModule` to imports array
