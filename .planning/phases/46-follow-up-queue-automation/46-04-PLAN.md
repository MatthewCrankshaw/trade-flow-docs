The processor class extends `WorkerHost` and implements:
```
async process(job: Job<IFollowupJobData>): Promise<void> {
  1. Extract { estimateId, revisionNumber, step } from job.data
  2. Re-read estimate: const estimate = await estimateRetriever.findByIdOrFail(estimateId)
  3. Check status: if estimate.status NOT in [SENT, VIEWED, RESPONDED, SITE_VISIT_REQUESTED]:
     - log("Follow-up skipped — estimate in status={status}", { estimateId })
     - return (silent no-op per FUP-06)
  4. Read document token: documentTokenRepository.findActiveByDocumentId(estimate.rootEstimateId, "estimate")
     - If no token found, log error and return (cannot build viewUrl)
  5. Read business: businessRetriever.findByIdOrFail(estimate.businessId) for businessName and ownerFirstName
  6. Build viewUrl from token: `${configService.get("APP_URL")}/estimate/${token.token}`
  7. Build priceDisplay from estimate.priceRange (formatRange utility or inline formatting)
  8. Map step to EstimateEmailSendType: { 1: FOLLOWUP_3D, 2: FOLLOWUP_10D, 3: FOLLOWUP_21D }
  9. Map step to followupStep: { 1: "3d", 2: "10d", 3: "21d" }
  10. Render HTML: estimateFollowupEmailRenderer.render({ businessName, followupStep, estimateNumber, viewUrl, traderFirstName, priceDisplay })
  11. Build subject: "Following up on estimate {estimateNumber} from {businessName}"
  12. Read recipient email from estimate (estimate.customer.email or via customer lookup — check what the estimate DTO provides)
  13. Send email: emailSenderService.sendEmail({ to: recipientEmail, subject, html, from: ... })
      - If sendEmail returns error, throw new Error to trigger BullMQ retry
  14. Create audit row: estimateEmailSendCreator.create({ estimateId, rootEstimateId: estimate.rootEstimateId, businessId: estimate.businessId, type: sendType, recipientEmail, subject, html, sentAt: DateTime.now() })
  15. Log success
}
```

The processor must NOT read recipient email from the job payload — it MUST read from MongoDB at execution time per FUP-06 (defence-in-depth, prevents stale payload from sending to wrong recipient).

Configure BullMQ retry: `attempts: 3, backoff: { type: "exponential", delay: 60000 }` via processor options or job-level config. Follow the STRIPE_WEBHOOKS pattern for retry configuration.

```typescript
@Processor(QUEUE_NAMES.ESTIMATE_FOLLOWUPS)
// OR: handled as job.name === "expiry" in a combined processor
export class EstimateExpiryProcessor extends WorkerHost {
  // ...
  async process(job: Job<IExpiryJobData>): Promise<void> {
    const { estimateId } = job.data;
    const estimate = await this.estimateRetriever.findByIdOrFail(estimateId);

    // Statuses eligible for auto-expiry
    const expiryWorthyStatuses = [
      EstimateStatus.SENT,
      EstimateStatus.VIEWED,
      EstimateStatus.RESPONDED,
      EstimateStatus.SITE_VISIT_REQUESTED,
    ];

    if (!expiryWorthyStatuses.includes(estimate.status)) {
      this.logger.log(`Expiry skipped — estimate in status=${estimate.status}`, { estimateId });
      return;
    }

    try {
      await this.estimateTransitionService.transition(estimateId, EstimateStatus.EXPIRED);
      this.logger.log("Estimate auto-expired", { estimateId });
    } catch (error) {
      this.logger.error(`Expiry transition failed for ${estimateId}`, (error as Error).stack);
      // Do not re-throw — job completes. Estimate stays in current status.
      // D-EXP-01: silent transition. If it fails, it fails quietly.
    }
  }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| BullMQ job payload -> processor | Untrusted data from Redis job enters processor logic |
| Processor -> email recipient | Processor determines who receives the follow-up email |
| Processor -> MongoDB write | Processor transitions estimate status |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-06 | Spoofing | Email recipient spoofing via stale job payload | mitigate | Processor NEVER reads recipient email from job.data. It re-reads the estimate from MongoDB at execution time to get the current customer email (FUP-06). Job payload only contains estimateId and revisionNumber as lookup keys. |
| T-46-07 | Tampering | Job payload poisoning with malicious estimateId | mitigate | Processor calls `EstimateRetriever.findByIdOrFail(estimateId)` which validates the ID exists and applies businessId policy check. A fabricated estimateId returns ResourceNotFoundError, aborting the job. |
| T-46-08 | Elevation | businessId policy bypass in processor | mitigate | EstimateRetriever.findByIdOrFail applies the standard EstimatePolicy.canRead check (business scoping). The processor does not bypass this — it uses the same retriever service that controllers use. The processor context has no authenticated user, so the policy check must be verified: if EstimateRetriever requires an authUser parameter, the processor must construct a service-level context or use an internal retriever that skips user-level auth but still validates business scoping. |
| T-46-09 | Tampering | Expiry processor transitions estimate without authorization | mitigate | EstimateTransitionService validates the transition is allowed in the ALLOWED_TRANSITIONS map. Attempting to transition from a terminal status (CONVERTED, DECLINED, etc.) fails with InvalidRequestError. The processor catches this and logs — no unauthorized state change occurs. |
</threat_model>
