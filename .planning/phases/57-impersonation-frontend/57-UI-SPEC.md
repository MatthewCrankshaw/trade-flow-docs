### Impersonation Banner Colors

Custom amber palette -- does NOT use design system semantic tokens. This is deliberate: the banner must be visually distinct from all application UI to prevent confusion.

| Element | Classes | Rationale |
|---------|---------|-----------|
| Banner background | `bg-amber-500` | High-contrast warning color, industry convention (Shopify, Stripe, AWS) |
| Banner text | `text-amber-950` | Maximum contrast against amber-500 background |
| Warning icon | `text-amber-950` | Matches text color |
| Return button | `bg-amber-950 text-amber-50 hover:bg-amber-900` | Inverted contrast for button prominence |

## Interaction Contracts

### Impersonate Button on User Detail Page

1. **Visibility**: Button appears only when ALL of the following are true:
   - The viewer has the `impersonate_user` permission
   - The target user is a customer user (has no support roles)
   - Impersonation is not already active
2. **Hidden entirely** when conditions are not met -- no disabled/greyed-out state (D-10).
3. **Button**: `Button variant="default"` with `UserCheck` icon (16x16, `mr-2`) and label "Impersonate".
4. **Placement**: In the user detail page header area, to the right of the user name/badges. On mobile, below the user name in the header stack.
5. **Click**: Opens `ImpersonateUserDialog`.

### Impersonation Confirmation Dialog

1. **Component**: `AlertDialog` (Radix).
2. **Title**: "Impersonate User".
3. **Description**: "You will see exactly what {name} sees, including their data, subscription, and permissions. All actions are logged."
4. **Reason field**: `Textarea` with `Label` "Reason", placeholder "Why are you impersonating this user?", required. Minimum 1 character after trim.
5. **Validation**: If reason is empty on submit, show inline validation message "A reason is required before impersonating a user." below the textarea in `text-sm text-destructive`. Action button remains enabled but submission is blocked until valid.
6. **Cancel button**: "Cancel" (`AlertDialogCancel`).
7. **Action button**: "Start Impersonation" (`AlertDialogAction`, default variant). Disabled during loading.
8. **On confirm**: Calls `POST /v1/impersonation/start` with `{ targetUserId, reason }` via RTK Query mutation.
9. **Loading state**: Action button shows spinner with "Starting..." text. Cancel button disabled during mutation.
10. **On success**:
    - Store impersonation token, target user info, and session ID in `impersonationSlice`.
    - Call `apiSlice.util.resetApiState()` to clear all cached data.
    - Navigate to `/` (customer dashboard root) to begin impersonated session.
    - Dialog closes.
11. **On error**: Dialog remains open. Error toast: "Unable to start impersonation. Please try again."

### Impersonation Banner

1. **Position**: `fixed top-0 left-0 right-0 z-50` -- above everything including the DashboardLayout header (z-20).
2. **Height**: 40px (`h-10`), single line.
3. **Background**: `bg-amber-500`.
4. **Layout**: `flex items-center justify-between px-4`.
5. **Left side**: `AlertTriangle` icon (16x16, `text-amber-950`) + `text-sm font-medium text-amber-950` "Impersonating: {user name}".
6. **Right side**: Button with label "Return to Support", styled `bg-amber-950 text-amber-50 hover:bg-amber-900 text-sm font-medium h-7 px-3 rounded-md`.
7. **Cannot be dismissed**: No close button, no click-away. Only "Return to Support" exits.
8. **Cannot be scrolled away**: Fixed positioning ensures it stays visible during scroll.
9. **Cannot be covered by modals**: z-50 sits above all overlays (Radix overlays use z-50 by default, so the banner should use `z-[60]` to guarantee visibility above modal backdrops).
10. **Content push**: When banner is active, `DashboardLayout` receives `pt-10` (40px) to prevent content overlap.
11. **Mobile responsive**: Same layout on mobile. Text truncates with `truncate` class if user name is long. Button text remains "Return to Support" (no abbreviation).

### Session Termination (Return to Support)

1. **Click "Return to Support"**: Calls `POST /v1/impersonation/end` with `{ sessionId }` via RTK Query mutation.
2. **On success**:
   - Clear `impersonationSlice` state (token, user info, session ID, active flag).
   - Call `apiSlice.util.resetApiState()` to clear all impersonated data from cache.
   - Navigate to `/support` (support dashboard).
   - Toast: "Impersonation session ended".
3. **On error**: Error toast: "Unable to end impersonation session. Please try again." Impersonation state remains active (user can retry).
4. **Loading state**: "Return to Support" button shows spinner, disabled during mutation. Banner remains visible.

### Expired Session Handling

1. **Detection**: In the RTK Query `baseQuery` wrapper, when an API response returns 401/403 with an impersonation-expired error code AND `impersonationSlice.active` is true, trigger the expiry flow.
2. **Expiry flow**:
   - Clear `impersonationSlice` state.
   - Call `apiSlice.util.resetApiState()`.
   - Navigate to `/support`.
   - Toast: "Impersonation session expired".
3. **No modal or acknowledgment required** (D-14). Automatic silent cleanup with a toast notification.

### Page Refresh Behavior

1. **Refresh ends impersonation** (D-03). Redux state is not persisted to sessionStorage or localStorage.
2. On refresh, `impersonationSlice` resets to initial state (active: false, no token).
3. Firebase auth resumes normally -- support user returns to their own support dashboard via the existing login redirect flow.
4. No error messages shown -- this is expected behavior, not an error state.

### Route Access During Impersonation

1. When `impersonationSlice.active` is true, the support user must be able to access business routes (Jobs, Customers, Quotes, Settings, etc.) that are normally behind `OnboardingGuard` and `PaywallGuard`.
2. The `getNavigationItems()` function checks impersonation state and returns customer business navigation items instead of support navigation items.
3. The sidebar renders the customer's navigation -- the support user sees exactly what the customer sees (D-07).
4. The `ImpersonationBanner` is the only visual indicator that this is not a normal customer session.

### prepareHeaders Token Switching

1. In `api.ts`, `prepareHeaders` checks `state.impersonation.active` first.
2. If active: use `state.impersonation.token` as the Bearer token.
3. If not active: use `auth.currentUser.getIdToken()` as normal (existing behavior).
4. No other changes to RTK Query configuration.

### Accessibility

1. `ImpersonationBanner` uses `role="alert"` and `aria-live="assertive"` to announce impersonation state to screen readers.
2. "Return to Support" button has `aria-label="End impersonation and return to support dashboard"`.
3. "Impersonate" button on user detail has `aria-label="Impersonate {name}"`.
4. Confirmation dialog uses Radix AlertDialog which provides modal focus trap and Escape key to cancel.
5. Reason textarea has associated `Label` via `htmlFor` attribute.
6. Banner has sufficient color contrast: amber-950 text on amber-500 background exceeds WCAG AA (contrast ratio > 7:1).
