**`src/auth/guards/permission.guard.ts`:** (per D-02, D-07, D-08)
- `@Injectable()` class `PermissionGuard implements CanActivate`
- Constructor takes `private readonly reflector: Reflector` (no repository needed -- simpler than SubscriptionGuard)
- `canActivate(context: ExecutionContext): boolean` method (synchronous, not async):
  1. Read metadata: `const requiredPermissions = this.reflector.getAllAndOverride<string[]>(PERMISSIONS_KEY, [context.getHandler(), context.getClass()])`
  2. If `!requiredPermissions || requiredPermissions.length === 0` return `true` (no restriction)
  3. Extract user: `const request = context.switchToHttp().getRequest()` then `const user = request.user as IUserDto | undefined`
  4. If `!user` throw `new ForbiddenError(ErrorCodes.ACTION_FORBIDDEN, \`Missing required permission: ${requiredPermissions.join(", ")}\`)`
  5. If `!hasAnyPermission(user, requiredPermissions)` throw same ForbiddenError
  6. Return `true`
- Import `PERMISSIONS_KEY` from `@auth/decorators/requires-permission.decorator`
- Import `hasAnyPermission` from `@auth/utilities/has-any-permission.utility`
- Import `ForbiddenError` from `@core/errors/forbidden-error.error`
- Import `ErrorCodes` from `@core/errors/error-codes.enum`
- Note: PermissionGuard is NOT registered as APP_GUARD (per D-09 anti-pattern). It is applied per-endpoint via `@UseGuards(PermissionGuard)` when needed in Phases 53-57.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client -> PermissionGuard | Untrusted request; user identity comes from JwtAuthGuard (upstream) |
| PermissionGuard -> request.user | User DTO with hydrated permissions; trusts JwtAuthGuard ran first |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-52-01 | Elevation of Privilege | PermissionGuard | mitigate | Guard returns `true` only when no metadata is set (endpoint has no @RequiresPermission); when metadata exists, user MUST have at least one required permission -- default-deny |
| T-52-02 | Elevation of Privilege | hasPermission | mitigate | Empty DtoCollection.some() returns false; null/undefined user throws ForbiddenError; no fallback to "allow" |
| T-52-03 | Spoofing | PermissionGuard | accept | Guard trusts request.user from JwtAuthGuard. JwtAuthGuard validates Firebase JWT with RS256 public keys -- spoofing requires compromising Firebase |
| T-52-04 | Information Disclosure | ForbiddenError message | accept | Error message reveals permission name ("Missing required permission: manage_users"). Per D-07, this is acceptable because the error handler obfuscates details in production |
</threat_model>
