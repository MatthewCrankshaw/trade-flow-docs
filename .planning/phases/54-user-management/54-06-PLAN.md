Fix all broken MongoDB $lookup join keys in the support module that cause subscription status filters, user detail business/subscription data, dashboard metrics, and role name resolution to return empty results.

Purpose: UAT Tests 4, 7, 9, and 10 all fail because $lookup pipelines compare user._id (MongoDB ObjectId) with subscription.userId (Firebase UID) -- different identifier systems that never match. Additionally, business $lookup references a non-existent $businessIds field, and role name $lookup projects the wrong field name.

Output: All $lookup joins corrected, unit tests updated, Firebase metadata service improved with initialization logging.
</objective>

- UserEntity has NO businessIds field -- only externalAuthUserId, email, nickname, name, privacySettings, supportRoleIds, businessRoleIds
- businessIds is a derived DTO field populated dynamically by UserRetriever querying the businessusers join collection
- BusinessUserEntity has businessId (ObjectId) and userId (ObjectId), stored in collection 'businessusers'
- SupportRoleEntity field is 'roleName' (not 'name')

  <action>
    Fix three broken $lookup pipelines in support-user.repository.ts:

    1. **findPaginated -- subscription $lookup (around line 72-75):**
       Change `let: { odUserId: { $toString: "$_id" } }` to `let: { odUserId: "$externalAuthUserId" }`.
       The pipeline expression `$eq: ["$userId", "$$odUserId"]` stays the same -- both sides are now Firebase UID strings.

    2. **findByIdWithDetails -- subscription $lookup:**
       Same fix as above. Change the `let` variable from `{ $toString: "$_id" }` to `"$externalAuthUserId"`.

    3. **findByIdWithDetails -- business $lookup:**
       The current $lookup references `$businessIds` which does not exist on the raw MongoDB user document.
       Replace with a two-step join:
       - First $lookup: Join `businessusers` collection where `businessusers.userId` equals the user's `_id` (both are ObjectIds). Use `let: { uid: "$_id" }` with `$expr: { $eq: ["$userId", "$$uid"] }`. Project only `businessId`.
       - $addFields: Extract the businessId values from the joined businessusers array into a flat array of ObjectIds.
       - Second $lookup: Join `businesses` collection using `$in` match against the extracted businessId array. Project `name` (and `_id`).
       - The final mapping should use the businesses array (not businessusers).

    4. **findByIdWithDetails -- supportroles $lookup projection (around line 159):**
       Change the projection from `{ name: 1 }` to `{ roleName: 1 }`.

    5. **findByIdWithDetails -- role name mapping (around line 214):**
       Change `r.name` to `r.roleName` in the mapping that extracts role names from the $lookup result.

    Update the unit test in support-user-retriever.service.spec.ts if the test mocks reference the old field names or join structures.

    Gap reasons:
    - Tests 4, 7: subscription join mismatch (externalAuthUserId vs ObjectId)
    - Test 7: business join references non-existent field
    - Test 10: role name projection uses wrong field
