## Convert to Quote Interaction Flow

Source: CONTEXT.md D-CONV-02, D-CONV-03, D-CONV-04, D-CONV-05.

### Flow sequence

1. Trader clicks "Convert to Quote" button
2. UUID v4 idempotency key generated in component state (not persisted to localStorage)
3. Button text changes to "Converting..." with `Loader2 animate-spin` icon, button disabled
4. `POST /v1/estimates/:id/convert` fires with `Idempotency-Key` header
5. On success: `toast.success("Estimate converted to quote")` then navigate to `/quotes/{quoteId}/edit`
6. On error: `toast.error("Failed to convert estimate. Please try again.")`, button re-enabled

### Button states

| State | Button |
|-------|--------|
| Ready | `<Button>Convert to Quote</Button>` with `ArrowRightLeft` icon |
| Converting | `<Button disabled>Converting...</Button>` with `Loader2 animate-spin` icon |
| Error | Button returns to Ready state |

### No confirmation dialog

The convert flow is not destructive -- it creates a new quote and transitions the estimate. No confirmation dialog needed. The mandatory review happens on the quote edit page (D-CONV-03).

## MarkAsLostDialog Layout Contract

Source: CONTEXT.md D-LOST-02, D-LOST-03, D-LOST-04.

### Dialog sizing

`sm:max-w-md` (compact dialog -- reason picker plus optional textarea).

### Structure

```
+------------------------------------------+
|  Mark estimate as lost?            [x]   |
|  This will lock the estimate and         |
|  cancel any pending follow-ups.          |
|                                          |
|  Reason (optional)                       |
|  +--------------------------------------+|
|  | [Too expensive]                      ||
|  | [Going with someone else]            ||
|  | [Decided not to do the work]         ||
|  | [Just getting an idea of costs]      ||
|  | [Timing isn't right]                 ||
|  +--------------------------------------+|
|                                          |
|  Anything else? (optional)               |
|  +--------------------------------------+|
|  |                                      ||
|  +--------------------------------------+|
|                                          |
|         [Keep Estimate]  [Mark as Lost]  |
+------------------------------------------+
```

### Reason chips

Same taxonomy as customer decline (Phase 45 D-CTA-04), same labels verbatim:

| Chip Label | API `reason` value |
|------------|-------------------|
| Too expensive | `too_expensive` |
| Going with someone else | `going_with_someone_else` |
| Decided not to do the work | `decided_not_to_do_work` |
| Just getting an idea of costs | `just_getting_idea` |
| Timing isn't right | `timing_not_right` |

- Layout: `flex flex-wrap gap-2`
- Single-select: tapping one deselects others. Tapping the selected chip deselects it (all optional per D-LOST-03).
- Unselected: `variant="outline"` with `text-sm`
- Selected: `bg-primary text-primary-foreground` with `text-sm`
- Touch target: `min-h-9` for comfortable tapping

### Freeform textarea

| Property | Value |
|----------|-------|
| Label | "Anything else? (optional)" in `text-sm font-semibold` |
| Placeholder | "Add a note if you'd like" |
| Max length | 500 characters |
| Character counter | `{count}/500` in `text-xs text-muted-foreground`, right-aligned, visible when text is entered |
| Rows | 3 |

### Confirm button states

| State | Button |
|-------|--------|
| Ready | `<Button variant="destructive">Mark as Lost</Button>` |
| Submitting | `<Button variant="destructive" disabled>Marking...</Button>` with `Loader2 animate-spin` |
| Keep Estimate | `<Button variant="outline">Keep Estimate</Button>` |

### Validation

Both reason and freeform text are optional (D-LOST-03). The confirm button is always enabled (no minimum input required). The API accepts `reason: null` and `notes: null`.

### Post-submit behavior

1. `toast.success("Estimate marked as lost")`
2. Dialog closes via `onOpenChange(false)`
3. RTK Query cache invalidation refreshes estimate detail -- status badge updates to "Lost", `EstimateLostReasonCard` appears, action buttons disappear

## State Contracts

### Loading State (Convert)

No dialog or skeleton needed. The "Convert to Quote" button shows a spinner during the POST. All other page content remains visible and interactive (except the action strip buttons which are part of the same component).

### Loading State (Mark as Lost)

Dialog opens immediately. Reason chips and textarea render instantly -- no async data fetch needed (reason taxonomy is a static constant).

### Error States

| Condition | Display | Copy |
|-----------|---------|------|
| Convert failure (server error) | Toast error | `toast.error("Failed to convert estimate. Please try again.")` |
| Convert failure (status invalid) | Toast error | `toast.error("This estimate cannot be converted in its current status.")` |
| Convert failure (network) | Toast error | `toast.error("Could not connect to the server. Check your connection and try again.")` |
| Mark as Lost failure (server error) | Toast error | `toast.error("Failed to mark estimate as lost. Please try again.")` |
| Mark as Lost failure (status invalid) | Toast error | `toast.error("This estimate cannot be marked as lost in its current status.")` |
| Mark as Lost failure (network) | Toast error | `toast.error("Could not connect to the server. Check your connection and try again.")` |

The MarkAsLostDialog remains open on error so the user can retry without re-entering data. The convert button returns to its ready state on error.

### Empty States

No empty states in this phase. Both flows operate on an existing estimate detail page which always has content.
