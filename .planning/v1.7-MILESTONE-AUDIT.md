### Critical Integration Gap: INT-01

**SubscriptionGuard blocks new user onboarding**

The `SubscriptionGuard` is registered as a global `APP_GUARD` in `app.module.ts` (line 59). Its logic:
- Passes through GET/HEAD unconditionally
- For POST/PATCH/DELETE: checks `findByUserId` for active/trialing subscription
- If no subscription exists → throws `ForbiddenError` (HTTP 403)

The new user onboarding sequence:
1. **PATCH /v1/user/me** (ProfileStep saves display name) → **BLOCKED** (no subscription, no `@SkipSubscriptionCheck`)
2. **POST /v1/user/me/business** (creates business) → **BLOCKED** (same reason)
3. **POST /v1/subscription/trial** (starts trial) → passes (`@SkipSubscriptionCheck` present)

**Impact:** Every new user who signs up is completely blocked at step 1 of onboarding. They cannot save their display name because SubscriptionGuard rejects the PATCH request before any subscription exists.

**Files to fix:**
- `trade-flow-api/src/user/controllers/user.controller.ts` — add `@SkipSubscriptionCheck()` to `patch()` method
- `trade-flow-api/src/business/controllers/business.controller.ts` — add `@SkipSubscriptionCheck()` to `create()` method

**Affected requirements:** TRIAL-01, ONBD-01, ONBD-02, ONBD-03, ONBD-04, ONBD-05, ONBD-06
